# SPEC-PERF-001: 인수 기준

## AC-001: 30분 녹음 전체 파이프라인 완료

**Given** 30분 길이의 m4a 녹음 파일이 존재할 때
**When** Flutter 앱에서 업로드하여 전체 파이프라인(STT → 화자분리 → 회의록 → 요약)을 실행하면
**Then** 모든 단계가 오류 없이 완료되고, 화자별 회의록과 요약이 정상 표시된다

- 각 단계의 상태가 `completed`로 전환됨
- 화자 분리 결과에 2명 이상의 화자가 식별됨
- 전체 처리 시간이 60분을 초과하지 않음

## AC-002: 화자 분리 청크 분할 동작

**Given** 30분 이상의 WAV 오디오 파일이 화자 분리 단계에 도달했을 때
**When** 화자 분리 처리가 시작되면
**Then** 오디오가 10분 단위 청크로 분할되어 각각 처리되고, 결과가 병합된다

- 청크 간 30초 오버랩이 적용됨
- 인접 청크의 동일 화자가 하나의 스피커 ID로 매핑됨
- 오버랩 구간의 중복 세그먼트가 제거됨

## AC-003: 30분 미만 녹음 기존 동작 유지

**Given** 10분 길이의 녹음 파일이 존재할 때
**When** 전체 파이프라인을 실행하면
**Then** 기존과 동일하게 단일 파일로 화자 분리가 처리된다 (청크 분할 없음)

- 기존 테스트가 모두 통과함
- 처리 결과가 청크 분할 전과 동일함

## AC-004: Redis visibility_timeout 설정

**Given** Celery 워커가 시작될 때
**When** Redis 브로커에 연결하면
**Then** `visibility_timeout`이 7200초(2시간)으로 설정된다

- `celery_app.py`에 `broker_transport_options` 설정 존재
- 60분 이상 소요되는 작업이 중복 실행되지 않음

## AC-005: 화자 분리 중간 진행률

**Given** 30분 녹음의 화자 분리가 3개 청크로 분할 처리될 때
**When** 각 청크 처리가 완료되면
**Then** Redis 상태가 중간 진행률로 업데이트된다

- 진행률이 0.30 → 0.47 → 0.63 → 0.80 순서로 업데이트됨
- Flutter 앱에서 진행률 변화가 폴링으로 감지됨

## AC-006: SoftTimeLimitExceeded 처리

**Given** Celery 작업에 `soft_time_limit=3600`이 설정되어 있을 때
**When** 작업이 60분을 초과하면
**Then** `SoftTimeLimitExceeded` 예외가 잡히고 부분 결과가 저장된다

- 작업 상태가 `failed`로 저장되며 "시간 초과" 메시지 포함
- 처리된 청크까지의 부분 결과가 Redis에 저장됨
- Flutter 앱에서 타임아웃 오류 메시지가 표시됨

## AC-007: 서버 재시작 스크립트

**Given** `deploy/restart.sh` 스크립트가 존재할 때
**When** 서버에서 `bash deploy/restart.sh`를 실행하면
**Then** uvicorn과 Celery 워커가 안전하게 재시작된다

- 기존 프로세스가 정리된 후 새 프로세스 시작
- 재시작 후 `curl localhost:8000/api/v1/health`가 200 응답

## 품질 기준

- 기존 테스트 전체 통과 (회귀 없음)
- 새로 추가된 코드에 대한 테스트 커버리지 85% 이상
- 30분 녹음 실제 E2E 테스트 통과
